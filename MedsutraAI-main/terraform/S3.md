# S3 Bucket Configuration

## Overview

This document describes the S3 bucket configuration for the AI Cancer Detection Platform. The infrastructure includes four S3 buckets designed for different purposes with appropriate security controls, encryption, and compliance features.

## Bucket Architecture

### 1. Medical Documents Bucket
**Purpose**: Store Protected Health Information (PHI) including patient reports, lab results, radiology images, and uploaded medical documents.

**Key Features**:
- **Encryption**: SSE-KMS using customer-managed key (`aws_kms_key.s3_encryption`)
- **Versioning**: Enabled for data recovery and audit trail
- **Access Logging**: All access logged to access logs bucket
- **Lifecycle Policy**: 
  - Old versions transition to STANDARD_IA after 30 days
  - Further transition to GLACIER after 90 days
  - Retention period: 7 years (2555 days) for compliance
- **Public Access**: Completely blocked
- **Bucket Policy**:
  - Denies unencrypted uploads
  - Enforces HTTPS/TLS for all connections
  - Enforces specific KMS key usage

**Compliance**: HIPAA-ready, DPDP Act compliant

**Naming**: `ai-cancer-detection-medical-docs-{account-id}`

### 2. Frontend Assets Bucket
**Purpose**: Store static website assets for the Next.js frontend application (HTML, CSS, JavaScript, images).

**Key Features**:
- **Encryption**: SSE-S3 (AES256)
- **Versioning**: Enabled for rollback capability
- **Access Logging**: All access logged to access logs bucket
- **Lifecycle Policy**:
  - Old versions deleted after 30 days
  - Incomplete multipart uploads aborted after 7 days
- **Public Access**: Blocked (access via CloudFront OAI only)
- **Bucket Policy**:
  - Enforces HTTPS/TLS for all connections
  - CloudFront Origin Access Identity (OAI) access only

**Naming**: `ai-cancer-detection-frontend-{account-id}`

### 3. Audit Logs Bucket
**Purpose**: Store application audit logs for compliance, security monitoring, and incident investigation.

**Key Features**:
- **Encryption**: SSE-S3 (AES256)
- **Versioning**: Enabled (required for compliance)
- **Access Logging**: All access logged to access logs bucket
- **Lifecycle Policy**:
  - Transition to GLACIER after 90 days
  - Transition to DEEP_ARCHIVE after 365 days
  - Retention period: 7 years (2555 days) for compliance
  - Old versions retained for 7 years
- **Public Access**: Completely blocked
- **Bucket Policy**:
  - Denies unencrypted uploads
  - Enforces HTTPS/TLS for all connections
  - Prevents log deletion (except by specific admin roles)
- **MFA Delete**: Should be enabled manually for additional protection

**Compliance**: HIPAA-ready, DPDP Act compliant (7-year retention)

**Naming**: `ai-cancer-detection-audit-logs-{account-id}`

### 4. Access Logs Bucket
**Purpose**: Store S3 access logs for all other buckets (meta-logging).

**Key Features**:
- **Encryption**: SSE-S3 (AES256)
- **Versioning**: Enabled
- **Lifecycle Policy**:
  - Transition to GLACIER after 90 days
  - Transition to DEEP_ARCHIVE after 365 days
  - Retention period: 7 years (2555 days)
- **Public Access**: Completely blocked

**Naming**: `ai-cancer-detection-access-logs-{account-id}`

## Security Features

### Encryption at Rest
- **Medical Documents**: Customer-managed KMS key (CMK) for maximum control
- **Other Buckets**: AWS-managed encryption (AES256) for cost optimization

### Encryption in Transit
- All buckets enforce HTTPS/TLS via bucket policies
- Any non-HTTPS request is denied

### Public Access Prevention
- All buckets have public access completely blocked
- Frontend bucket accessible only via CloudFront OAI
- Medical documents and audit logs have no public access

### Versioning
- All buckets have versioning enabled
- Protects against accidental deletion
- Provides audit trail for compliance

### Access Logging
- All buckets (except access logs bucket) log access to the access logs bucket
- Provides comprehensive audit trail
- Supports security monitoring and incident investigation

### Bucket Policies
Each bucket has a least-privilege bucket policy:
- **Medical Documents**: Enforces KMS encryption, denies unencrypted uploads
- **Frontend Assets**: Enforces HTTPS, allows CloudFront access
- **Audit Logs**: Prevents deletion, enforces encryption, HTTPS-only

## Compliance Features

### HIPAA Readiness
- Encryption at rest and in transit
- Access logging and audit trails
- Versioning for data integrity
- 7-year retention for audit logs

### DPDP Act Compliance
- Encryption of PHI data
- Access controls and logging
- Data retention policies
- Ability to delete patient data (versioning allows recovery if needed)

### ABDM Alignment
- Secure storage for ABHA-linked patient records
- Audit trails for data access
- Encryption standards aligned with Indian healthcare regulations

## Cost Optimization

### Bucket Keys
- All buckets have bucket keys enabled
- Reduces KMS API calls by 99%
- Significant cost savings for high-volume buckets

### Lifecycle Policies
- Automatic transition to cheaper storage classes
- Medical documents: STANDARD → STANDARD_IA → GLACIER
- Audit logs: STANDARD → GLACIER → DEEP_ARCHIVE
- Reduces storage costs while maintaining compliance

### Intelligent Tiering
- Consider enabling S3 Intelligent-Tiering for medical documents bucket
- Automatically moves objects between access tiers based on usage patterns

## Post-Deployment Configuration

### MFA Delete for Audit Logs Bucket
MFA Delete provides additional protection against accidental or malicious deletion. This must be configured manually using AWS CLI:

```bash
# Enable MFA Delete (requires root account)
aws s3api put-bucket-versioning \
  --bucket ai-cancer-detection-audit-logs-{account-id} \
  --versioning-configuration Status=Enabled,MFADelete=Enabled \
  --mfa "arn:aws:iam::{account-id}:mfa/root-account-mfa-device {mfa-code}"
```

### CloudFront Origin Access Identity
After creating CloudFront distribution (future task), update the frontend assets bucket policy to allow CloudFront OAI access.

### IAM Roles and Policies
Create IAM roles with least-privilege access to buckets:
- **Lambda Execution Role**: Read/write access to medical documents bucket
- **EKS Service Account**: Read access to medical documents bucket
- **Application Role**: Write access to audit logs bucket

## Monitoring and Alerts

### CloudWatch Metrics
Monitor the following S3 metrics:
- `BucketSizeBytes`: Track storage growth
- `NumberOfObjects`: Monitor object count
- `AllRequests`: Track API request volume
- `4xxErrors` and `5xxErrors`: Monitor access errors

### CloudWatch Alarms
Set up alarms for:
- Unusual access patterns to medical documents bucket
- Failed access attempts (403/404 errors)
- Rapid growth in bucket size
- Lifecycle transition failures

### S3 Access Analyzer
Enable S3 Access Analyzer to:
- Identify buckets with public access
- Detect overly permissive bucket policies
- Monitor cross-account access

## Backup and Disaster Recovery

### Cross-Region Replication (Optional)
For disaster recovery, consider enabling cross-region replication:
```hcl
resource "aws_s3_bucket_replication_configuration" "medical_documents" {
  bucket = aws_s3_bucket.medical_documents.id
  role   = aws_iam_role.replication.arn

  rule {
    id     = "replicate-medical-documents"
    status = "Enabled"

    destination {
      bucket        = aws_s3_bucket.medical_documents_replica.arn
      storage_class = "STANDARD_IA"
    }
  }
}
```

### Backup Strategy
- **Versioning**: Provides point-in-time recovery
- **Lifecycle Policies**: Automatic archival to GLACIER/DEEP_ARCHIVE
- **Cross-Region Replication**: Optional for disaster recovery
- **AWS Backup**: Consider using AWS Backup for centralized backup management

## Troubleshooting

### Common Issues

**Issue**: Access denied when uploading to medical documents bucket
- **Cause**: Missing KMS key permissions or incorrect encryption header
- **Solution**: Ensure IAM role has `kms:GenerateDataKey` permission and upload uses correct KMS key

**Issue**: CloudFront cannot access frontend assets bucket
- **Cause**: Missing OAI permissions in bucket policy
- **Solution**: Update bucket policy to allow CloudFront OAI access

**Issue**: Lifecycle transitions not working
- **Cause**: Insufficient permissions or incorrect lifecycle rule configuration
- **Solution**: Verify IAM permissions and lifecycle rule syntax

**Issue**: Access logs not appearing
- **Cause**: S3 log delivery group lacks permissions
- **Solution**: Ensure access logs bucket policy allows S3 log delivery
